# Homeground V1 — 可执行实现简报

版本：2026-07-18
状态：当前实施依据；保留 `homeground-v1-inquiry-handoff-spec.md` 作为完整研究记录，不把其中的自建 CRM 设计作为本轮上线前提。

## 1. 这一版要证明什么

Homeground 的第一件事不是自动卖行程，而是完成一条真实、诚实的路径：

```text
四问得到起始路线
  → 看懂这只是起始方向，不是可订行程
  → 用唯一入口联系工作室
  → 人工确认真实旅行条件
  → 向合适的当地服务方询价 / 核验
  → 再给具体规划和价格
```

成功不是“用户点了按钮”，而是工作室能够收到一条带路线上下文的真实咨询，并由人开始下一步沟通。

## 2. 供应能力带来的边界

现有地接资料表明：华东有可询价的同行资源；张家界主要是成本参考；黄金游轮只有模板行程与计费规则；没有能支撑实时全国库存或固定价格的统一供应网络。

因此公开页面只能承诺：

> Homeground 先理解旅行条件，再人工核对适合的当地服务。价格、库存、船期、交通和服务范围都在确认后才会说明。

页面不得承诺：

- 已核实的交通、酒店、门票、舱位或地接库存；
- 全国自有地接团队、固定净价或最低价；
- 免签资格或签证结果；
- 自动报价、自动生成可执行日程或直接预订。

## 3. V1 的用户体验

### 保留

- 四题路线生成器：同行者、旅行偏好、总晚数、节奏。
- 先给城市、晚数、取舍与未确认条件，后索要联系。
- 英文、中文、韩文的同一条路线逻辑。
- 路线结果页中的一个、且只有一个联系入口。

### 现在不加

- 第五题“城市多选”；不会选中国城市的游客不应先做地理考试。
- 自动把城市插入路线；固定地点先作为人工需要复核的上下文。
- 价格计算器、报价、付款、合同、自动 PDF、在线聊天机器人或 Codex SDK。
- 自建 CRM、复杂的重复线索判定、运营者角色/审批/SLA 引擎。

### 联系入口

联系入口位于路线结果之后，先说明：这不是预订；日期、机场、季节、交通和具体服务尚未核验。

入口只收集一条可选说明，例如：已订航班、固定城市、同行者需求、去过的地方或必须避免的安排。它不会自动修改路线。

正常入口是网页内的单一咨询表单：用户填写自己的 Email 和选填说明，服务端保存成功后，网页才显示“已收到”。工作室随后从 Gmail 直接回复这个用户邮箱。

预填邮件只允许在服务端明确确认“没有保存”时作为应急通道；打开邮件应用不能被说成“咨询已提交”。WhatsApp 只有完成企业账号与跨境收发测试后才考虑开放。

## 4. 人工工作流

收到路线咨询后，人工第一条回复只需要补齐实际询价所必需的信息：

1. 日期与是否可浮动；
2. 抵达 / 离开城市与已订交通；
3. 成人、儿童及儿童年龄；
4. 希望使用的语言；
5. 酒店和房型舒适度、粗略预算；
6. 必去 / 不想去的城市和活动。

然后由工作室根据目的地和服务类型，向适合的当地服务方核对可行性与价格。对客路径保持简单；内部只需 Gmail 中的 `Homeground inquiries` 标签、未读状态和已发送邮件。两个人按时段轮流值班即可。

## 5. 分两步实现

### 第一步：前端与人工工作台

- 结果态的 Header、Studio 与页尾 CTA 统一指向 `#planner-handoff`，不再把用户送回已看过的题目。
- 移除正常流程中的长 `mailto:` 路由；加入一个收集客户 Email 和选填说明的联系 handoff。
- 只有服务端确认保存成功后才显示收到；明确失败时才出现简短应急邮件。
- WhatsApp 暂不显示；电话号码和跨境收发能力通过真实测试后再评估。
- 不在浏览器储存、分析或伪造保存任何联系方式 / 说明。

### 第二步：真实 Inquiry 上线配置（不能假装已经完成）

需要用户/工作室提供：Supabase 与 Resend 正式账号、数据存储地区、隐私联系人、保留期限和真实回复时效。

现有实现包含：服务端校验过的表单、保存成功后的参考编号，以及投递到当前 Gmail 的通知。完成正式账号配置与真实收发测试后才能打开生产开关。WhatsApp 只在 WhatsApp Business 及跨境实测通过后开放为回复方式。

## 6. 上线前语言与信任校验

- 三种语言只能承诺团队实际能回复的语言；不能默认韩语人工服务已经到位。
- 免签页不进入导航或 sitemap；现有资料不是可对外发布的签证依据。
- 没有真实评价就不用评价模块；团队框架可保留，不用在首页放人像。
- GA 等非必要分析应在取得同意后才加载；拒绝不影响路线与联系。

## 7. 验收标准

- 四问和现有路线结果不改变；
- 任何已有结果时，主 CTA 都抵达同一个 handoff；
- 手机上不横向溢出，键盘可以完成所有交互；
- 联系内容不会进入 `sessionStorage`、站内 URL 或分析；
- 只有服务端保存成功才显示已收到；任何文案都不声称已经预订；
- 联系消息包含路线、晚数、四项回答和用户的可选说明；
- 后续服务端版本必须用真实收件箱做端到端测试，再显示成功状态。
